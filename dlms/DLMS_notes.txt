Interfaces with the DLMS to MuleESB and ZMQ:

we have 4 communication interfaces right now. we need firstly to describe them in RAML format then I will generate Java pojos from that description that those can be used inside java as DTO's.

1. DLMS-UG API:
DLMS generates the data transfer cost value and sends it to the Utility Generator to further model the final optimisation problem instance (inside the CP-Generator).
BUT, the UG can also ask so the UG-->DLMS is the request or command flow. DLMS-->UG is data (data transfer cost) flow.

2. Adapter-->DLMS API: 
Adapter will coordinate with the DLMS for the data-related reconfigurations (such as data migrations). The Adapter component triggers an update which further leads to a reconfiguration of the cost function evaluation process.
 
3. Monitoring:
DLMS extracts the monitoring data (from Monitoring Plane --> ESPER via MuleESB) about the current system state (such as data transfer rate). Not decided yet.

4. Data Catalog/MetaData API:
The registered data will be modelled into the Model repository. The Data Catalog API shall be responsible to load the required metadata-related information into the DLMS.
